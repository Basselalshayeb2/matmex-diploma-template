\section{Особенности реализации}
\label{sec:solution}

В данном разделе описывается реализованный прототип подсистемы голосового управления для МИС «СТОММИС».
Решение построено как конвейер: на стороне клиента в браузере выделяются речевые сегменты, а на сервере выполняются транскодирование (при необходимости), распознавание речи (ASR), затем фильтрация по транскрипции с использованием бинарного классификатора \textit{command/not-command}, и только после этого --- интерпретация команды в контролируемое действие внутри МИС.


\subsection{Клиентский модуль: захват аудио и VAD}
\label{subsec:task1}

Клиентская часть реализована как веб‑страница, получающая доступ к микрофону через Web Audio API \cite{WebAudio}.
Далее непрерывно выполняется детекция речевой активности (VAD) с использованием \texttt{@ricky0123/vad-web} \cite{RickyVADWeb}, исполняющего нейросетевую модель в формате ONNX через ONNX Runtime Web \cite{ONNXRuntimeWeb}.
VAD выдаёт события начала и окончания речи, а также оценку вероятности речи.
Пример событий VAD-модели:
\begin{itemize}
    \item \texttt{onSpeechStart}~--- фиксируется начало речевого сегмента;
    \item \texttt{onSpeechEnd}~--- сегмент считается завершённым после паузы (периода «тишины») заданной длительности;
    \item \texttt{onUpdate}~--- обновление вероятности речи (для визуализации и отладки).
\end{itemize}

Для завершения команды используется правило: \enquote{конец команды --- 2 секунды тишины}.
Практически это реализуется параметром ожидания «тишины» после окончания речи (например, \texttt{redemptionFrames}, эквивалентный приблизительно двум секундам).

Пороговые параметры VAD настраиваются экспериментально под условия кабинета:
\texttt{positiveSpeechThreshold} — порог включения речи (старт сегмента при превышении),
\texttt{negativeSpeechThreshold} — порог выключения (переход в тишину),
а \texttt{redemptionFrames} задаёт сколько кадров тишины подряд требуется для фиксации окончания сегмента.

\subsection{Клиентский модуль: выделение сегментов речи}
\label{subsec:task2}

Основная проблема непрерывного распознавания речи состоит в том, что на сервер может передаваться нерелевантная информация (общение с пациентом, фоновые реплики), что увеличивает нагрузку и стоимость обработки.

В предлагаемом решении клиент выполняет только выделение сегментов речи с помощью VAD, не принимая решения о том, является ли сегмент командой.
Каждый завершённый речевой сегмент передаётся на сервер в виде аудиофайла.

Дальнейшее решение о необходимости обработки сегмента принимается на сервере после распознавания речи.
Такое разделение ответственности позволяет упростить клиентскую часть и перенести более сложную логику фильтрации в серверный контур, где доступны текстовые данные и вычислительные ресурсы.

\subsection{Wake word: варианты реализации и выбранный подход}
\label{subsec:wakeword}

В рамках работы были реализованы и проверены два подхода.

\textbf{1) Wake word в браузере (локальный KWS).}
Прототип выполнял обнаружение ключевого слова на клиенте по аудиопризнакам (на уровне коротких сегментов, выделенных VAD).
Практические испытания показали две проблемы:
\begin{itemize}
    \item высокая доля пропусков (false negatives) в реальных условиях кабинета;
    \item необходимость обучения/калибровки под каждого врача (разные голоса и микрофоны), что неудобно для внедрения.
\end{itemize}
Поэтому данный вариант не был принят как основной.

\textbf{2) Wake word на сервере (проверка по ASR).}
В итоговой версии используется серверная проверка: короткий сегмент, выделенный VAD, распознаётся ASR, после чего выполняется текстовая проверка на наличие ключевого слова (например, \enquote{старт}).
Это увеличивает число обращений к ASR в режиме ожидания, но снижает риск пропусков и не требует обучения под каждого врача.
Фактически режим ожидания работает как \enquote{VAD + короткие ASR‑проверки}, а после успешной активации следующий сегмент речи рассматривается как команда.

Практические испытания показали, что регулярное использование wake word неудобно для врача в условиях реального приёма и снижает естественность работы.
Поэтому в итоговой версии прототипа данный механизм рассматривается как резервный сценарий и базовая линия сравнения, тогда как основной подход использует автоматическую идентификацию команд по транскрипции после ASR.

\subsection{Серверный модуль: приём аудио, ASR и интерпретация}
\label{subsec:task3}

Серверная часть реализована на Python (Flask) и предоставляет HTTP‑endpoint для приёма командных аудиосегментов.
Клиент отправляет запрос вида:
\begin{itemize}
    \item метод \texttt{POST};
    \item тело \texttt{FormData} c полем \texttt{file};
    \item дополнительные поля метаданных (идентификатор врача, пациента, режим команд), если требуется контекст.
\end{itemize}

На сервере выполняются шаги.
\begin{enumerate}
    \item проверка наличия поля \texttt{file} и чтение байтов аудио в память;
    \item приведение формата аудио к поддерживаемому входу ASR: если MIME‑тип отличается от ожидаемого, выполняется транскодирование в Ogg Opus через \texttt{ffmpeg} по пайпам (без записи на диск)~\cite{FFmpeg};
    \item распознавание речи посредством Yandex SpeechKit~\cite{YandexSpeechKit};
    \item применение бинарного классификатора \textit{command/not-command} к полученной транскрипции;
    \item при классификации как \textit{command} --- интерпретация результата: преобразование текста в структурированную команду (например, \texttt{intent} + набор параметров), пригодную для заполнения полей карты пациента.
\end{enumerate}

Интерпретация реализуется гибридно: для короткого списка команд задаётся допустимый набор интентов, а для сопоставления свободных формулировок используется LLM‑модуль.
Результат возвращается клиенту в JSON и/или применяется на стороне МИС «СТОММИС».

\subsection{Серверная идентификация команд по транскрипции}
\label{subsec:text_classifier}

После распознавания речи (ASR) необходимо определить, является ли полученная транскрипция командой для системы или относится к речевому мусору.
Для этого используется бинарный классификатор \textit{command/not-command}.

В реализации классификатор построен на текстовых признаках транскрипции.
Используются символьные n-граммы (char n-grams), извлекаемые с помощью TF-IDF, что обеспечивает устойчивость к ошибкам распознавания речи.
Дополнительно учитываются признаки предметной области, такие как наличие номеров зубов, специфических медицинских терминов и характерных глаголов действий.

Модель классификации реализована как логистическая регрессия с балансировкой классов, что позволяет учитывать несбалансированность данных (команд существенно меньше, чем обычной речи).

Выбор бинарной постановки задачи обусловлен следующими причинами:
\begin{itemize}
    \item задача фильтрации «команда/мусор» существенно проще полной интерпретации;
    \item модель может быть лёгкой и быстро выполняться;
    \item классификатор позволяет сократить число вызовов LLM и тем самым уменьшить вычислительную нагрузку и стоимость обработки.
\end{itemize}

При выборе порога решения приоритетом является высокая полнота (recall) класса \textit{command}, чтобы минимизировать пропуски команд врача.

Обобщённая логика серверной обработки может быть представлена следующим образом:
\begin{verbatim}
    audio -> ASR -> text
    if not is_command(text):
        stop
    else:
        result = LLM(text)
        validate(result)
        apply_to_STOMMIS(result)
\end{verbatim}

\subsection{Ограничение вывода языковой модели и валидация результата}
\label{subsec:llm_constraints}

Для интерпретации распознанного текста команды используется большая языковая модель
\texttt{qwen3-235b-a22b-fp8}, доступная через совместимый с OpenAI API интерфейс
(\texttt{base\_url = https://rest-assistant.api.cloud.yandex.net/v1}, \texttt{project = folder\_id}).
Задача модели заключается не в генерации произвольного текста, а в преобразовании транскрипции команды
в структурированное действие, пригодное для выполнения в МИС «СТОММИС».

Чтобы исключить некорректные или опасные действия, вывод модели жёстко ограничивается:
\begin{itemize}
    \item допускается только формат \textbf{JSON} фиксированной структуры;
    \item тип действия выбирается только из заранее определённого множества интентов (белый список);
    \item параметры команды (слоты) проверяются на корректность типов и принадлежность к допустимым значениям;
    \item при неоднозначности модель обязана вернуть статус \texttt{needs\_clarification} вместо выполнения действия.
\end{itemize}

\paragraph{Структура результата (контракт).}
Результат интерпретации команды представляется объектом следующего вида:
\begin{verbatim}
{
  "intent": "set_diagnosis | set_tooth_state | add_procedure | ...",
  "slots": { ... },
  "confidence": 0.0..1.0,
  "needs_clarification": true|false,
  "clarification_question": "..."
}
\end{verbatim}

\paragraph{Схема допустимого вывода.}
Для обеспечения воспроизводимости и проверки корректности используется схема (упрощённый вариант):
\begin{verbatim}
intent: enum[ set_diagnosis, set_tooth_state, add_procedure, cancel, ... ]
slots:  object (зависит от intent)
confidence: number [0..1]
needs_clarification: boolean
clarification_question: string (optional, only if needs_clarification=true)
\end{verbatim}

\paragraph{Пример интерпретации команды.}
Пусть врач произносит: \textit{«Зуб 26: кариес. Добавь пломбу.»}.
После распознавания речи система получает текстовую транскрипцию и передаёт её в LLM вместе с контекстом
(текущий пациент, выбранный врач, активная карта, доступные справочники).

Ожидаемый результат интерпретации:
\begin{verbatim}
{
  "intent": "set_tooth_state",
  "slots": {
    "tooth": "26",
    "state": "caries",
    "procedure": "filling"
  },
  "confidence": 0.83,
  "needs_clarification": false
}
\end{verbatim}


\paragraph{Выбор и конфигурация языковой модели.}
В прототипе для семантической интерпретации команд используется большая языковая модель \texttt{qwen3-235b-a22b-fp8}, доступная через API облачной платформы (вызов через совместимый клиент OpenAI с заданным \texttt{base\_url}).
Выбор данной модели обусловлен качеством понимания русскоязычных команд и устойчивостью к вариативным формулировкам при сохранении приемлемой задержки.
Параллельно проводится исследование альтернативной модели \texttt{gpt-oss-20b}; на момент написания работы результаты сравнения являются предварительными, поэтому в основной конфигурации прототипа фиксируется модель \texttt{qwen3-235b-a22b-fp8}.

\paragraph{Ограничение допустимого вывода модели.}
Чтобы исключить неконтролируемые действия, в прототипе применяется управляемая генерация, при которой поведение LLM контролируется через три канала:
\textbf{(1) инструкцию} (системное сообщение), \textbf{(2) ввод} (контекст, добавляемый к пользовательской команде), и \textbf{(3) инструменты} (tool-вызовы для получения допустимых значений из предметной области).
Сервер принимает результат \emph{только} при успешной валидации JSON по схеме и принадлежности значений разрешённым множествам.

\subparagraph{Инструкция (правила работы и формат вывода).}
Системная инструкция фиксирует назначение модели, запрещает свободный текст и задаёт формат результата согласно схеме из предыдущего пункта.
Упрощённый фрагмент инструкции:

\begin{verbatim}
    Ты - часть процедуры обработки данных первичного осмотра пациентов. Тебе будут приходить запросы вида:
    1. Команда пользователя (распознанная и переведенная в текст речь врача при осмотре)
    2. Пронумерованный список вариантов обработки этой команды (в том виде, который понятен информационной системе)

    Необходимо в соответствие с командой, полученной от врача, выбрать наилучшие варианты ее обработки из списка и вернуть их список в структурированном виде,
    соответствующим следующей JSON схеме:
    {
        "$schema": "http://json-schema.org/draft-07/schema#",
        "type": "array",
        "items": {
            "type": "object",
            "properties": {
                "chosen_option": { "type": "integer" },
                "V": { "type": "string" },
                "ABC": { "type": "string" },
                "Z": { "type": "string" },
                "to_delete": { "type": "integer" }
            },
            "required": ["id", "chosen_option", "V", "ABC", "Z", "to_delete"],
            "additionalProperties": False       #
        }
    }
\end{verbatim}

\subparagraph{Инструменты (tool) для зубной карты.}
Для сценариев заполнения зубной формулы LLM получает доступ к инструменту, возвращающему допустимые действия и параметры
для конкретного клинического контекста (например, для выбранного зуба).
Инструмент реализует поиск по доменному индексу допустимых команд и возвращает структурированные данные,
которые используются моделью для выбора \texttt{intent} и заполнения \texttt{slots}.

\subsection{Интеграция с МИС «СТОММИС» и особенности предметной области}
\label{subsec:stommis}

В отличие от решений, ориентированных на «протоколирование» консультации, в рассматриваемом сценарии цель~--- заполнение структурированной карты.
Это требует:
\begin{itemize}
    \item приведения распознанного текста к заранее определённым категориям (диагноз, манипуляция, зуб, рекомендация);
    \item сопоставления со справочниками (например, список процедур и заболеваний), используемыми в МИС «СТОММИС»;
    \item минимизации количества действий врача в интерфейсе.
\end{itemize}

Командный режим позволяет врачу произносить короткие инструкции, которые конвертируются в конкретные изменения в карте.
Например, команда \enquote{старт, кариес на 26 зубе} может быть интерпретирована как выбор диагноза \enquote{кариес} и установка соответствующего зуба в формуле.

\subsection{Нефункциональные требования: ресурсы}
\label{subsec:nfr}

Решение спроектировано с учётом ограничений клиники:
\begin{itemize}
    \item \textbf{Ресурсоэффективность:} постоянная работа VAD в браузере должна быть существенно дешевле постоянного ASR;
    \item \textbf{Минимизация трафика:} на сервер отправляются только короткие сегменты команд;
    % \item \textbf{Приватность:} исключается постоянная передача полного аудио приёма на сервер; объём передаваемых данных ограничивается командными сегментами.
\end{itemize}

Перечисленные свойства проверяются экспериментально в разделе~\ref{sec:experiment}.
